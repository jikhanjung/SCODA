# SCODA 요약: 자기완결형 데이터 아티팩트

## 왜 필요한가

현재 학술 데이터는 대부분 중앙집중식 데이터베이스와 웹 서비스를 통해 배포된다. 이 서비스 중심 모델은 항상 "최신 상태"를 우선시하며, 과거 데이터는 묵시적으로 덮어쓰여진다. 이로 인해 세 가지 핵심 문제가 발생한다.

- **재현성 훼손**: 분석에 사용한 데이터가 이후 변경되면 동일한 결과를 재현할 수 없다.
- **책임성 모호**: 데이터가 점진적으로 갱신되면 특정 해석에 대한 책임 소재가 불분명해진다.
- **해석 가능성 저하**: 데이터가 원래의 맥락과 출처로부터 분리되면 올바르게 해석하기 어렵다.

특히 분류학처럼 동일한 데이터에 대해 여러 정당한 해석(*sensu*)이 공존하는 분야에서, 하나의 "정답"만 유지하는 서비스 모델은 학술적 다원성을 근본적으로 수용하지 못한다. 기존 표준(Frictionless Data, RO-Crate, BagIt)은 버전 관리나 불변성을 *표현*할 수는 있으나 *강제*하지는 않으며, 도구와 워크플로는 여전히 현장 갱신을 기본으로 가정한다.

## 어떻게 구현되는가

SCODA는 데이터셋을 **불변이고, 버전 관리되며, 인용 가능한 아티팩트**로 배포한다. 하나의 아티팩트는 다섯 가지 필수 구성 요소를 포함하는 단일 파일(예: SQLite)이다.

| 구성 요소 | 역할 |
|-----------|------|
| **데이터** | 데이터셋 자체 (외부 의존 없이 자체 포함) |
| **정체성** | 고유 식별자 + 의미론적 버전 (예: trilobase v1.0.0) |
| **의미론** | 스키마, 필드 설명, 표시 의도 등 해석에 필요한 정보 |
| **출처** | 원천 문헌, 빌드 과정, 큐레이터 정보 |
| **무결성** | SHA-256 해시 및 선택적 디지털 서명 |

운영 구조는 세 계층으로 나뉜다. **정규 서버**가 최신 뷰와 검색을 제공하고, **SCODA 릴리스**가 특정 시점의 불변 스냅샷을 보존하며, **로컬 오버레이**가 원본을 훼손하지 않으면서 개인 주석과 대안적 해석을 지원한다. 변경이 필요하면 기존 아티팩트를 수정하는 것이 아니라 새 버전을 생성한다.

또한 **표시 의도(display intent)** 메커니즘을 통해 데이터가 어떻게 보여져야 하는지(트리, 테이블, 그래프 등)를 아티팩트 안에 선언하여, 전용 도구 없이도 즉시 탐색할 수 있도록 한다. 이는 4계층(표시 의도 → UI 매니페스트 → 저장된 쿼리 → UI 에셋)으로 구성되어 뷰어 역량에 따라 우아하게 퇴화된다.

## 어떤 유용성이 있는가

- **정확한 인용**: 특정 버전의 데이터 상태를 정확히 참조할 수 있어 학술 논문에서의 데이터 인용이 명확해진다.
- **완전한 재현성**: 과거 분석에 사용된 정확한 데이터를 언제든 복원할 수 있다.
- **오프라인 사용**: 서버 가용성에 의존하지 않고 단일 파일만으로 데이터를 활용할 수 있다.
- **해석적 다원성 지원**: 여러 학술적 견해를 하나의 프레임워크 안에서 공존시키고 비교할 수 있다.
- **즉각적 해석 가능성**: 표시 의도를 통해 낯선 데이터셋도 별도 문서 없이 바로 탐색 가능하다.
- **기존 표준과의 호환**: Frictionless Data, RO-Crate, BagIt 등을 SCODA 아티팩트 내부에 통합할 수 있어, 기존 생태계를 대체하지 않고 거버넌스 계층을 추가한다.

논문은 삼엽충 분류 데이터베이스 Trilobase를 SCODA 아티팩트로 구조화한 사례를 통해, 이 모델이 실제 학술 데이터에 자연스럽게 적용됨을 보여준다.
